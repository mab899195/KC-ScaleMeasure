# Direct Page Jump Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Let users jump to any page in a multi-page PDF by clicking the page-number indicator and typing a number, instead of clicking prev/next repeatedly.

**Architecture:** Replace the static `<span id="pcur">` in the pager with an `<input id="pcur">` styled to look identical to the surrounding text. Wire up Enter (jump), Escape (cancel), blur (cancel), and focus-selects-all behaviors. The existing `renderPage(n)` function does the actual page rendering — we just call it with the parsed value.

**Tech Stack:** Single-file vanilla HTML/JS/CSS app (`index.html`). PDF.js for rendering. No build step, no test framework — verification is manual in browser.

**Spec:** `docs/superpowers/specs/2026-05-19-page-jump-design.md`

### Task 2: Wire up the input behavior

**Files:**
- Modify: `index.html` (JS block — the `renderPage` function around line 242, and add handlers near the existing `$('pprev').onclick` line 260)
- Modify: `index.html` (the `$('f').onchange` handler around line 263)

- [ ] **Step 1: Update `renderPage` to write to `.value` instead of `.textContent`**

In `index.html`, find this line in `renderPage()`:

```js
$('pcur').textContent = n;
```

Replace with:

```js
$('pcur').value = n;
if(document.activeElement === $('pcur')) $('pcur').blur();
```

The blur ensures that after a successful jump triggered by Enter, focus returns to the body so the global arrow-key handler resumes working.

- [ ] **Step 2: Set the input width when a PDF loads**

In `$('f').onchange`, find this line:

```js
$('ptot').textContent = pdfDoc.numPages;
```

Add immediately after it:

```js
$('pcur').style.width = (String(pdfDoc.numPages).length + 1) + 'ch';
```

This sizes the input to fit the largest possible page number plus one character of breathing room, so the pager doesn't reflow as the user navigates.

- [ ] **Step 3: Add focus, keydown, and blur handlers for `#pcur`**

After the existing pager wiring (after line 261 where `$('pnext').onclick` is defined), add:

```js
$('pcur').onfocus = () => $('pcur').select();
$('pcur').onblur = () => { $('pcur').value = curPage; };
$('pcur').onkeydown = e => {
  if(e.key === 'Enter'){
    e.preventDefault();
    const n = parseInt($('pcur').value, 10);
    if(Number.isFinite(n) && n >= 1 && n <= pdfDoc.numPages && n !== curPage){
      renderPage(n);
    } else {
      $('pcur').value = curPage;
      $('pcur').blur();
    }
  } else if(e.key === 'Escape'){
    $('pcur').value = curPage;
    $('pcur').blur();
  }
};
```

Notes for the implementer:
- `onfocus` selecting all means clicking the input lets the user immediately overtype the current value.
- `onblur` reverts the displayed value to `curPage` so a user who types `7` but clicks away without pressing Enter doesn't see a stale `7` while still on page 1.
- The arrow-key global handler at line 350-351 already guards with `if(e.target.tagName==='INPUT') return;` — confirm this line exists; it means arrow keys move the cursor inside the input rather than navigating pages.

- [ ] **Step 4: Manually verify in the browser**

Reload `index.html` and load a multi-page PDF (10+ pages ideal). Run through the spec's testing checklist:

1. Load a single-page PDF → pager hidden, no input visible.
2. Load a multi-page PDF → pager visible, input shows `1`, input width visually fits N digits.
3. Click the input → contents highlight (selected).
4. Type `7`, press Enter → page 7 renders. Prev button enables, next button stays enabled. Focus returns to body (test by pressing right arrow → page advances to 8).
5. Click input, type `999`, press Enter → reverts to current page, no navigation.
6. Click input, type `abc`, press Enter → reverts to current page.
7. Click input, type `5`, press Escape → reverts, no navigation.
8. Click input, type `5`, click elsewhere → reverts, no navigation.
9. Click input, press Left arrow → cursor moves in input, page does NOT change.
10. After successful jump, press Right arrow with no input focus → page advances.

If any check fails, debug before committing. Common gotchas:
- If clicking the input doesn't select contents, check `onfocus` wiring.
- If arrow keys page-navigate while typing, verify the existing line `if(e.target.tagName==='INPUT') return;` is intact in the `window` keydown handler.
- If focus stays on input after Enter and arrow keys move the cursor instead of pages, verify the `.blur()` call was added in `renderPage`.
